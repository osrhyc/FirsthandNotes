> TL;DR：第 3 篇把金字塔原理用于咨询报告和复杂问题分析。核心不是一上来写解决方案，而是先界定问题：现状是什么，期望结果是什么，差距在哪里，导致差距的结构是什么。解决问题类文章通常回答三类问题：应该做什么，是否应该做，应该怎么做。序言负责定性问题，正文负责展示分析步骤或理由。

## 详读

### 先界定问题

作者认为，问题不是“我想讨论一个主题”，而是现状和目标之间存在差距。现状是非期望结果，目标是期望结果。解决方案就是从现状走向目标的方法。

这个定义很实用，因为它能防止文章变成泛泛分析。比如“如何提高阅读效率”太空；更清楚的问题是：“我读了很多书，但写作时找不到可用材料，如何把阅读转化为可复用笔记？”

后一种问题有现状、目标和差距，也更容易构建金字塔。

### 三类常见问题

第 3 篇指出，解决方案类文章常常回答三类问题：

| 读者状态 | 文章要回答 |
| --- | --- |
| 不知道解决方案 | 我们应该做什么？ |
| 已经有一个方案 | 我们应该做吗？ |
| 已经接受方案 | 我们应该怎么做？ |

这对写学习文档很有帮助。不同读者状态，文章结构应该不同。新手需要“应该做什么”；管理者可能需要“是否值得做”；执行者需要“怎么做”。

### 分析先于表达

这一篇强调，复杂报告不能只靠表达技巧。你必须先有条理地收集和分析数据，把问题结构化，再把结果转成金字塔。

作者给出的理想流程可以概括为：

```text
界定问题 -> 结构化分析 -> 找到解决方案 -> 组成金字塔表达
```

如果前两步没做，后面只是把混乱材料包装得更好看。

### 分析框架的作用

作者承认复杂问题通常模糊、混乱、数据多、解决方案多，所以需要诊断框架和逻辑树。框架不是为了装饰，而是为了避免遗漏关键要素。

对这个项目来说，可以把它转成内容拆解模板：

| 场景 | 先问 |
| --- | --- |
| 学习文档 | 新手从哪里卡住？目标能力是什么？ |
| 行业事件 | 当前状态和预期状态的差距是什么？ |
| 读书笔记 | 作者要解决什么问题？解决方案是什么？ |
| 产品分析 | 用户现状、目标和障碍分别是什么？ |

## 案例档案

| 案例 | 一句话经过 | 在书中的作用 |
| --- | --- | --- |
| 销售/库存报告问题 | 原始问题被拆成数据、格式和时效问题 | 展示如何从罗列转为结构化问题 |
| 咨询报告流程 | 先界定问题，再分析，再表达 | 说明复杂报告不能跳过诊断阶段 |

## 要点清单

- 问题是现状和目标之间的差距。
- 解决问题类文章先判断读者要问哪一类问题。
- 复杂文章要先分析，再表达。
- 逻辑树和诊断框架用于避免遗漏。
- 序言定性问题，正文展示分析和答案。
