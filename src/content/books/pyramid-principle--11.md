> TL;DR：问题＝你已有的结果（R1，非期望结果）与你想要的结果（R2，期望结果）之间的差距，而差距不是凭空来的，它长在一段特定背景里。界定问题就是回答三个问题：发生了什么事（切入点/序幕＋困扰／困惑）、我们不喜欢它什么（R1）、我们想要什么（R2）。四要素排好之后，按「从左到右再往下」的原则，把读者最后了解的那个事实放进冲突，序言的 S-C-Q 就自动出来了。读者的疑问共有 7 种形态，取决于解决方案走到了哪一步——还没想出来、想出来不确定对不对、对了不会实施、实施了不管用、有好几个选不出来、说不清 R2、不知道自己是否在 R1。作者用「连续分析」的 5 个问题收口：问题 1、2 的答案是序言，问题 3~5 的答案是金字塔里的思想。

## 详读

### 问题的定义：R1 与 R2 之间的差距

本章开篇给了一个极简的定义：判断问题是否存在，看经过努力得到的结果（现状）与希望得到的结果（目标）之间是否有差距。由某一特定背景导致的某一特定结果，作者称为**非期望结果 R1**（Undesired Result，即现状）；你不喜欢这个结果、想要别的结果（比如销售额从降低变成增长），那个想要的就是**期望结果 R2**（Desired Result，即目标）。所谓解决方案，就是「如何从 R1 到 R2」（图 8-1、8-4）。

这个界定方法作者称为**连续分析**（统计学上叫序列分析），它把解决问题拆成按逻辑顺序排列的 5 个问题：是否有问题（或机会）？问题在哪里？为什么存在？我们能做什么？我们应该做什么？作者给出的归类（表 8-1）很关键：问题 1、2 是**界定问题**，问题 3 是**找原因**，问题 4、5 是**找最佳方案**。而在介绍分析结果时——问题 1、2 的答案就是文章的序言，问题 3~5 的答案是金字塔结构里的思想、观点、论点、看法。这一句把第三篇和前两篇接了起来。

### 界定问题的框架：问题从背景里长出来

作者强调，差距「不是凭空产生的」，它来自某一背景，在一系列特定条件下产生。所以了解发展历史，是确定差距性质、把握其重要性的最基本过程。

用一个设例展开（图 8-2）：假如你有一家公司，30 年来靠一套屡试不爽的方法卖工业不动产——销售人员列潜在客户名单、写推销信、邮寄给所有潜在客户。销售额每年增长约 10%，但今年第四季度种种迹象表明销售将**减少 10%** 而不是增加 10%。

把它装进框架：**背景**由切入点/序幕组成，即现有的结构和流程（常规销售方法），这个流程本该导致 R2（保持 10% 增长）；但背景中发生了某事、或采取了某一行动（即预测销售额），有可能导致 R1（销售增长低于预期）。寄信后的结果与期望结果之间的差距，就是问题。

于是界定问题的框架要你回答 3 个问题：

- 发生了什么事情？［背景（切入点/序幕 + 困扰／困惑）］
- 我们不喜欢它什么？（R1）
- 我们想要什么？（R2）

作者接着指出一个容易被跳过的要点：**原因通常可以从序幕所描述的活动中找到**，解决方案也从改变现有结构和流程入手。所以销售额降低的可能原因是：名单已过期、和/或推销信没说服力、和/或邮寄效率低——三条全部落在序幕描述的那套流程上。下一步才是建立诊断框架和逻辑树做结构化分析（第 9 章）。

### 四要素的展开：序幕、困扰、R1、R2

作者说要确定 4 个要素才能界定问题，并建议把它们当**戏剧术语**理解：切入点/序幕（Starting Point/Opening Scene）、困扰／困惑（Disturbing Event）、R1、R2。

**切入点/序幕**：想象你坐在黑暗的剧场里，大幕拉开，舞台布景描绘某一特定时刻的某一特定地点——这就是序幕；某一事件的发生使剧情展开——那就是困扰。界定问题同理，大幕开启后你看到的是自己或客户所在的企业/行业问题出在哪里。序幕由结构或流程组成（表 8-2）。作者给的写法示范很朴素：说「以前有一家公司从 3 个仓库向全国配送家庭用品……」，读者自然得到 3 个仓库配送的印象（图 8-8）；说「我们公司由许多独立经营的业务部门组成，每一项业务都与图像处理的新技术有关」，读者得到另一幅图（图 8-9）。原则：序幕阶段「尽量作简单的设想和简短的描述」，等到写序言时再展开（「展开说明框架中的各要素」小节）。

**困扰／困惑**：指现在发生、即将发生或未来将发生的事件，它威胁序幕中那个相对稳定的背景，并因此引发 R1。上面那个例子里，新技术的出现就是困扰。来源分三类——**外部原因**（新竞争对手、改用新技术、政府或消费政策变化）、**内部原因**（增加业务流程、装新计算机系统、进军新市场、调整产品线）、**近期认识到的其他原因**（产品/流程性能落后、运营水平低于平均、市场研究显示消费者态度可能变化）。作者还留了个现实出口：写咨询项目建议书时信息不充分，你可能确定不了是什么让你意识到问题存在，但你总能找到客户对现有结构或流程不满意的地方。

**R1**：读者需要设法解决、或有可能面临的问题，也可能是有可能抓住的机会。通常由困扰引起。在咨询业，R1 是客户前来咨询的主要动因——但作者提醒，客户可能并不清楚导致 R1 的真正原因。困扰更可能带来的是：对结构或流程产生负面影响、扰乱某一特定方面的工作、引发对业务/产品/流程的重新考虑、挑战关于客户/市场/竞争/核心竞争力/流程或技术的基本假设。R1 可以不止一个，示意图里用尽可能简短的文字写。

**R2**：作者在这里下了一个硬要求——「只有对 R2 进行尽可能具体和定量的描述，才能判别你取得的结果是否是期望结果」。没有全面准确的 R2 描述，你在思考过程中就很难选择方案。示范写法：实现全年增长目标；将产品上市时间缩短 1/3；让超市收取的试销费用合理；调整系统以保证正常运行；有足够的生产能力满足预测的需求。实在描述不了最终结果时，就在 R2 里写下「如果问题得到解决，你希望达到的状态」——然后解决问题的第一步就是把 R2 具体化。

作者最后补了一句诚实话：序幕、困扰、R1、R2 的界定**可能在解决问题的过程中变化**（比如收集资料时对外部变化认识更清楚了，就要重新提炼 R1 和 R2），但不管各部分怎么变，**要素之间的相互关系始终占主导地位**。

### 把「界定的问题」写成序言：从左到右再往下

框架的最大好处是能很容易地转换成序言（图 8-3）。规则只有一条：**遵循从左到右再往下的原则，把读者最后了解的事实作为序言的冲突部分，并在此基础上提出疑问**。

但作者立刻承认工业不动产那个是高度简化的例子，现实要复杂得多：

- 企业已经发现问题并有了解决方案 → 问题变成「该方案是否正确」或「如何实施该方案」，**解决方案本身成了引发问题的冲突**（图 8-5）。
- 有方案但发现不可行或没效果 → 问题变成「我们该做什么」（图 8-6）。
- 甚至可以是**三个层级**（图 8-7）。作者的设例：层级一，你是大型包装食品生产商，超市尚处新兴阶段，你想让新品上市前在超市货架试销一周，超市怕打乱秩序不同意，你同意付试销费后他们接受了；层级二，超市发展成连锁，试销费涨到每周 **2 万美元**，你认为难以容忍，组织委员会调查，除了拒付想不出别的办法；层级三，超市也拒绝了为你的产品做一周试销。故事很复杂，但展开逐条分析后，序言里寥寥数句就能说清——技巧仍然是从左到右再往下，把读者了解的最后事实当冲突：背景 S 收进层级一、二的全部 R1/R2/解决方案，冲突 C 只放层级三的 R1（超市拒绝试销），疑问 Q＝我们应该如何应对面临的问题。

作者由此给出使用框架的操作步骤：展开问题的基本部分；判断你的解决方案处于哪一阶段（已经提出了，还是已经被接受了）；提出适当的疑问；检查序言是否呈现了界定的问题；检查金字塔是否回答了疑问。

### 发掘读者的疑问：7 种情形

下一步是找读者的疑问。作者点名一个「严重错误」：写文章时不能确定读者是否已经采取了措施——而知道读者何时采取措施、该措施如何影响文章要回答的疑问，会大大简化序言写作和后续推理。

按寻找解决方案的初衷不同，读者通常想解决 7 种问题之一（图 8-10）：

最常见的三种——(1) 不知道如何从 R1 到 R2；(2) 知道如何从 R1 到 R2，但不敢肯定是否正确；(3) 知道正确方案，但不知道如何实施。变形两种——(4) 方案已实施但由于某种原因行不通；(5) 确定了好几个方案，不知选哪一个。可能但不常见的两种——(6) 知道 R1，但不能具体描述 R2，所以无法找方案；(7) 知道 R2，但不清楚自己是否处在 R1（作者标为「这是典型的标杆比对」）。

对应的序言写法（图 8-11 到 8-17，全文在附录 2「序言结构范例」）作者给了抽象骨架：

- **(1) 我们应该做什么**：S＝系统目前的工作情况，C＝它不做它该做的工作，Q＝如何让它做它该做的工作？关键词是「变革」。另一种是告诉别人如何做某事，关键词是「步骤」：S＝这是我们准备开展的活动，C＝我们没能力开展该活动，Q＝我们如何有能力开展该活动？
- **(2) 是否应该做想做的事**：S＝我们存在问题，C＝我们计划采取行动，Q＝该行动是否正确？变形：S＝我们计划采取 X 行动，C＝除非出现 Y 情况否则不希望采取 X，Q＝Y 情况会出现吗？
- **(3) 应该如何做想做的事**：S＝我们曾经存在问题，C＝通过 X 行动我们解决了问题，Q＝你是如何进行 X 行动的？或：S＝我们有一个目标，C＝我们正在安装一套系统或流程来实现该目标，Q＝该系统或流程是如何运行的？
- **(4) 方案行不通该怎么办**：是第一种结构的延伸，问题同样是「我们应该做什么」，区别在于已经采取过两三次措施，**需要追溯过去的历史才能发现真正的问题**。
- **(5) 应该选哪种方案**：备选方案一般归在冲突项下。作者在这里下了一条很硬的忌讳——不要一一列出所有备选方案（除非读者事先已知），否则读者会误把它们当成选择某方案的原因；尤其要避免罗列后逐个否定，形成「A 方法不好→B 方法不好→因此采用 C 方法」的关键句要点。理由一句话说透：采用 C 的原因其实并不是 A 和 B 不好，而是 C 行之有效。
- **(6) 应该采用哪些战略**：客户面临问题或机会、觉得该行动，但因不熟悉情况或知识有限，不知道怎么定目标和措施（比如身处技术与市场变化很快的行业，认为多变局面正是从不景气行业脱身进入成长性行业的机会，却不知进入后会变成什么样）。于是请顾问分析行业、确定成功关键因素、自身优势、竞争成功率与盈利能力，制定最明智的战略。战略是文章的主要思想，关键句要点描述制定战略的步骤，也可用归纳法（上面方框是结论，下面 3 个方框是步骤）。
- **(7) 我们存在问题吗**：文章关注处于变化中的行业，最典型的场景是客户希望瞄准同行业竞争对手，或其他行业中采取同样措施的企业。

### 实战案例：家庭用品零售商的仓储能力

作者用一个现实中家庭用品零售商的例子收尾（图 8-18、8-19）。要素如下：公司有 **3 个配送中心**，分别位于**伍斯特、伊凡斯维尔、拉斯韦加斯**，另外还从 **DMSI 公司**租场地做配送中心。3 个仓库的设计能力可供应 **490 家商店**，但事实上 4 个中心供应现有的 **438 家商店**有时都很紧张。若供货量年增长率 **4%~5%**，加上明年年底前计划开张的 **198 家新店**，预计**两年后仓储能力将不够用**。

公司确定可采取的方案包括：扩建现有仓库；新建一到两个仓库；改进商品处理流程；继续依赖第三方。不同方案对 **ROI** 影响不同。公司希望选一种战略，既保证最低的投资和运营成本，又不改变现有的处理速度和全品种战略——这就是被具体化了的 R2。

序言结构是图 8-10 中**第五种情形的变型**：S＝我们存在问题，C＝我们有不同的解决方案，Q＝选择哪个方案？

作者最后坦承框架「对初读者而言，接受和理解起来会有困难」，但无论口头还是书面解释问题它都特别有用；并且明确边界：**从界定问题到找到解决方案，还要经过对问题的实际分析**——分析原因、评估各种方案。框架的价值在于提高找原因和做结构化分析的效率，具体怎么做留给第 9 章「结构化分析问题」。

## 延伸案例（编者补）

编者补，非原书内容。本章的 7 种读者疑问在今天的工作场景里最容易踩的是第 (5) 种——「应该选哪种方案」。常见写法是在文档里先摆出「我们有三个选项」，再逐个论证 A 不行、B 不行、所以选 C。按明托本章的判断，这个结构会让读者把「A 和 B 不好」误读成「选 C 的原因」，而真正的原因应该是 C 行之有效。补一条实践上的对照：如果读者事先并不知道 A 和 B 存在，把它们摆进来反而是在给自己制造需要反驳的论点。

另补一条关于第 (7) 种情形的说明：作者把「知道 R2 但不清楚自己是否处在 R1」标为标杆比对（benchmarking）的典型场景，这也解释了为什么这类项目往往从「同行是怎么做的」开始——它本质上是在替客户确认 R1 是否成立。

## 怎么用

1. 动笔前先把四要素写成四行：序幕（现有的结构/流程是什么）、困扰（发生了什么事威胁到它，且注明是外部/内部/近期认识到的哪一类）、R1（我们不喜欢它什么）、R2（我们想要什么）。R2 那一行如果写不出数字或具体最终结果，就先别写方案——按作者的说法，解决问题的第一步是确定具体的 R2。
2. 写序言时执行「从左到右再往下」：把读者**最后**了解的那个事实放进冲突 C，其余全部压进背景 S，疑问 Q 从 C 自然长出来。层级越多，压进 S 的越多——包装食品那个例子里两整个层级的 R1/R2/方案都进了 S。
3. 下笔前对着 7 种情形先定位读者站在哪一格：还没方案（1）／有方案不确定（2）／方案对但不会做（3）／做了不管用（4）／方案太多（5）／说不清 R2（6）／不确定自己是否有问题（7）。定位不同，序言和金字塔结构完全不同；而且要特别确认读者**是否已经采取过措施**——作者把这一点漏判称为作者们犯的严重错误。

## 要点清单

- 问题＝R1（现状/非期望结果）与 R2（目标/期望结果）之间的差距；解决方案＝如何从 R1 到 R2。
- 差距不是凭空产生的，它来自背景；了解发展历史是确定差距性质和重要性的最基本过程。
- 连续分析 5 问：问题 1、2 的答案是序言，问题 3~5 的答案是金字塔里的思想。
- 界定问题只需回答 3 问：发生了什么事（序幕＋困扰）、我们不喜欢它什么（R1）、我们想要什么（R2）。
- 四要素是戏剧术语：序幕是布景，困扰是让剧情展开的那件事，R1 是它引发的结果，R2 是你想要的结果。
- 困扰分三类来源：外部原因、内部原因、近期认识到的其他原因。
- R2 必须尽可能具体和定量，否则无法判别取得的结果是不是期望结果，也无法在方案间做选择。
- 原因通常藏在序幕描述的活动里——所以解决方案从改变现有结构和流程入手。
- 序言转换只有一条规则：从左到右再往下，读者最后了解的事实即冲突。
- 读者的疑问有 7 种形态，取决于解决方案走到哪一步；漏判读者是否已采取措施是严重错误。
- 备选方案归在冲突项下，不要罗列后逐个否定——选 C 的原因是 C 行之有效，不是 A、B 不好。
- 框架各要素的界定会在解决问题过程中变化，但要素之间的相互关系始终占主导地位。
